Transformer的使用——pipline的三步：tokenizer的预处理，输入传入模型，后处理

  • 使用tokenizer进行预处理

    • 将文本转换成模型可以接受的形式——tokenizer

      1. 将输入拆分为称为标记的单词、子单词或符号（如标点符号）
      2. 将每个标记映射到一个整数
      3. 添加可能对模型有用的额外输入

    • 所有这些预处理都需要以与模型预训练时完全相同的方式完成——AutoTokenizer类及from_pretrained()

      1. 使用我们模型的checkpoint名称，它会自动获取与模型的标记器关联的数据并将其缓存

    • 将句子传给分词器，将得到一个字典——是一个包含两个键的字典，input_ids和attention_mask，接下来要把input_ids列表转换成tensor形式（TensorFlow只接受tensor形式的输入）

      1. 指定想要返回的张量类型（pytorch、TensorFlow或普通的numpy）——return_tensors

  • 遍历模型

    • 可以使用tokenizer一样的方法下载预训练模型，transformers提供了一个AutoModel类及from_pretrained()方法

    • Transformers模块的输出通常有三个维度

      1. Batch size: 一次处理的序列数（在我们的示例中为2）。

      2. Sequence length: 序列的数值表示的长度（在我们的示例中为16）。

      3. Hidden size: 每个模型输入的向量维度。

        由于最后一个值，它被称为“高维”。隐藏的大小可能非常大（768通常用于较小的型号，而在较大的型号中，这可能达到3072或更大）

  • 模型头

    Head一般是由若干层的线性层来构成的。

    Transformers库中的主要模型架构有：

    • *Model (retrieve the hidden states)
    • *ForCausalLM
    • *ForMaskedLM
    • *ForMultipleChoice
    • *ForQuestionAnswering
    • *ForSequenceClassification
    • *ForTokenClassification
    • …

    单纯的*Model，就是不包含 Head 的模型，而有For*的则是包含了具体 Head 的模型。

    例如，对于前面的那个做在情感分析上pretrain的checkpoint(distilbert-base-uncased-finetuned-sst-2-english)，我们可以使用包含 SequenceClassification 的Head的模型去加载，就可以直接得到对应分类问题的logits，而不仅仅是文本向量表示

  • Postprocessing the output

    • 现在的输出不是概率，只是logits（最后一层输出的原始未归一化分数）要通过softmax层和loss function

    • 为了获得每个位置对应的标签，可以检查模型配置的id2label属性

Models

  • creating a Transformer

    • 初始化BERT——加载配置对象configuration object

  • 不同的加载方式

    • 从默认配置创建模型会使用随机值对其进行初始化，因此加载已经训练过的Transformers模型——from_pretrained()方法

      模型的存储地址：~/.cache/huggingface/transformers

  • 保存模型

    • 保存模型类似加载模型（from_pretrained()）那样——save_pretrained()

      会得到两个文件config.json（模型架构所需的属性） 和 pytorch_model.bin（就是state dictionary，包含模型所有的权重）

  • 使用transformers模型进行推理

    • 对于模型的输入

      1. Transformer模型只能处理tokenizer生成的numbers

      2. tokenizer把输入转换成列表形式的词汇索引（input ids）

      3. 虽然模型接受许多不同的参数，但只需要 input IDs

    • Tokenizer

      1. Word-based:将原始文本拆分为单词并为每个单词找到一个数字表示,每个单词都被分配了一个 ID，从 0 开始一直到词汇表的大小。该模型使用这些 ID 来识别每个单词。——one-hot

      2. character-based：中文

      3. subword tokenization：依赖于不应该将频繁使用的词拆分成更小的子词，而将稀有词分解成有意义的子词的原则——黏合性语言（例如土耳其语）

    • Tokenizer的加载与保存

      1. 同样是用from_pretrained()和save_pretrained()

    • 文本Encoding(input_ids的生成)

      encoding有两步：tokenization、转换成input_ID

      1. tokenization：分词成token
      2. convert_tokens_to_ids：将token转化为number，构建一个tensor传给模型（tokenizer有一个词汇表）

      tokenization：分词成token

      convert_tokens_to_ids()：从tokens到IDs

    • Decoding——从词汇索引中，我们想要得到一个字符串

      当我们使用预测新文本（从提示生成的文本，或用于翻译或摘要等序列到序列问题）的模型时

  • 问题：处理多个序列

    • 处理多个序列？

      上面会报错的原因是转换成tensor之后增加了一个维度

      因此重新加一个维度

      Batching 是一次通过模型发送多个句子的行为

    • 处理多个序列不同长度

      批处理的时候，将两个句子组合在一起时，他们的长度可能不同，而tensor必须是矩阵，因此无法将input_IDs直接转换为tensor——*填充输入

    ​

    同样一句话单独和在句子里得到的logit完全不同，这是因为Transformer的注意力，这里是填充token，为了在通过模型传递不同长度的单个句子时，或者在传递一批应用了相同句子和填充的句子时获得相同的结果，我们需要告诉这些注意层忽略填充标记——attention mask

    Attention masks是与输入ID张量形状完全相同的张量，用0和1填充：1s表示应注意相应的标记，0s表示不应注意相应的标记

    这样加了attention_mask的，对于同一句子得到的logit是相同的

    • 序列太长

      1. 使用支持的序列长度较长的模型：Longformer、LED

      2. 截断序列:指定max_sequence_length

  • Tokenizer的封装

    ==在sentence上直接调用tokenizer，就得到可以直接输入模型的input==

    对于调用tokenizer这种转换，会对分词的结果，在开头添加特殊词[CLS]，在末尾添加特殊词[SEP]，因为预训练用到这些

    • 可以根据几个目标进行填充
    • 可以截断序列
    • 转换不同架构的tensor，tokenizer默认返回是np，但模型的输入必须是tensor

训练一个batch的分类器

  • 从Hub加载数据集https://huggingface.co/datasets

    • MRPC：里面都是一对一对句子

      默认情况下位于*~/.cache/huggingface/datasets*中

    • 可以使用raw_datasets通过索引来访问每对句子

    • 获取每个整数对应的label，可以调用raw_train_dataset的features返回每类的类型

    • 使用分词器处理数据集

    • 如果是要判断两个句子是否有相同的含义，不能只将两个序列传给模型，应该将两个序列作为一对来处理，并进行预处理

      有两个句子时，模型期望输入是[CLS] sentence1 [SEP] sentence2 [SEP]的形式

    • token_type_ids是区分哪个是第一个句子哪个是第二个，如果选择不同的检查点，可能在tokenized inputs中没有返回token_type_ids（DistilBERT 就不会返回）只有在预训练中见过的时候模型才知道返回他们

    • 使用tokenizer处理整个数据集

      但这样有缺点，返回是字典（字典的键是input_ids ， attention_mask 和 token_type_ids），只有转换过程有足够的内存来存储整个数据集时才不会出错

    • 将数据保存为数据集——Dataset.map()

      这样预处理就不仅仅是tokenizer，因此定义一个函数作用在数据集的每个元素上

      上面没有用到padding参数，因为在标记的时候将所有样本填充到最大长度的效率不高。一个更好的做法：在构建批处理时填充样本更好，因为这样我们只需要填充到该批处理中的最大长度，而不是整个数据集的最大长度。当输入长度变化很大时，这可以节省大量时间和处理能力

  • 动态填充（collate function）——DataCollatorWithPadding

    将元素批处理在一起后，要做的最后一件事是将所有实例填充到最长元素的长度

    得到的8个数据的长度，动态填充就是要把这个batch里的长度都填充成67

使用 Trainer API

Transformers 提供了 Trainer class帮助进行模型的fine-tuning。一旦已完成了数据预处理，则只需要简单的几步，就可以使用 Trainer 进行模型的训练了

  • 数据预处理

  • Training

    默认情况下，Trainer和TrainingArguments会使用：

    • batch size=8
    • epochs = 3
    • AdamW优化器

  • 评估——compute_metrics()函数的构建

    预测用trainer.predict，predict()输出是带有三个field的命名元组

    1. predictions：预测结果，得到的logit
    2. label_ids
    3. metrics：包含loss和一些时间指标（预测时间，总时间，平均时间）也会包含计算完compute_metrics()的值

    把预测结果predictions转换成和label一样再比较，所以需要取2轴上的最大值（因为它是二维的）

    转换之后，使用load_metric()加载MRPC数据集的指标，使用compute函数来计算指标

    将以上包装在一起得到compute_metrics()函数
